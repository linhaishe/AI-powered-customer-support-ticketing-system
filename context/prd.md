# 天猫海外 AI 客服服务中台 PRD

## 1. 产品名称

天猫海外 AI 客服服务中台。

## 2. 产品定位

天猫海外 AI 客服服务中台是一个面向跨境电商客服场景的 AI 全栈业务系统。它模拟接入天猫海外现有的订单、物流、售后、商家、平台规则和客服工单体系，为平台客服提供工单摘要、业务上下文聚合、风险识别、处理建议、知识引用和多语言回复草稿。

第一阶段以作品集和简历项目为目标，优先完成一个可演示、可解释、可继续迭代的客服工作台 Demo。系统先使用 mock 数据和本地 AI 辅助函数完成完整业务闭环，后续可演进为接入真实 AI、数据库、缓存、队列和知识检索的全栈项目。

## 3. 项目目标

### 业务目标

- 降低客服理解复杂跨境工单的时间成本。
- 将订单、物流、售后、商家和规则信息聚合到同一工作台。
- 通过 AI 提供可编辑、可追踪、可人工确认的处理建议。
- 强调 AI 安全边界，避免 AI 直接执行退款、赔付、售后关闭等高风险动作。

### 简历目标

- 展示从 0 到 1 的业务产品设计能力。
- 展示复杂业务对象抽象和全栈架构设计能力。
- 展示 AI 在真实业务流程中的落地方式，而不是只做通用 Chatbot。
- 展示前端工程、服务端聚合、数据建模、AI 结构化输出和风险控制意识。

## 4. 目标用户

### 平台客服

处理消费者咨询、投诉和售后问题，需要快速理解订单、物流、售后和平台规则上下文。

### 客服主管

关注高风险工单、客服处理效率、AI 建议采纳率和服务质量。

### 平台运营

关注高频问题、物流异常、商家服务风险、知识库缺口和规则解释争议。

### 商家运营

关注商品问题、售后原因、商家响应质量和消费者反馈趋势。

## 5. 核心场景

### 主场景：物流清关停滞导致消费者投诉

消费者反馈订单超过预计送达时间，物流轨迹显示包裹已到达目的国，但清关状态多日未更新。消费者情绪不满，希望获得明确解释、预计送达时间或补偿方案。

客服需要完成：

- 查看消费者原始问题和历史沟通。
- 查看订单、物流、售后和商家上下文。
- 判断是否属于高风险投诉。
- 查找平台关于跨境物流和清关时效的说明。
- 生成谨慎、准确、符合消费者语言偏好的回复。
- 必要时升级给物流专员或主管复核。

AI 需要辅助：

- 总结消费者诉求。
- 标记问题类型为物流延迟或清关异常。
- 判断情绪和风险等级。
- 给出处理建议。
- 引用平台规则和物流说明。
- 生成多语言回复草稿。

### 次要场景

- 消费者咨询退款进度。
- 消费者询问关税或清关费用。
- 消费者反馈商品描述与实物不一致。

## 6. MVP 范围

### 第一阶段必须完成

- 首页改造成客服工作台。
- 提供 3 到 4 条模拟跨境服务工单。
- 支持工单切换。
- 展示消费者问题、历史沟通、订单、物流、售后、商家和规则上下文。
- 展示 AI 摘要、问题类型、情绪、风险等级、处理建议、知识引用和回复草稿。
- 支持 AI 草稿的采纳、编辑、忽略交互。
- 使用 TypeScript 类型约束业务对象。
- 使用 mock 数据模拟天猫海外现有业务系统。
- 使用本地 AI 辅助函数模拟结构化 AI 输出。

### 第一阶段不做

- 登录和权限系统。
- 真实数据库。
- 真实 AI API 调用。
- 真实天猫海外内部系统接入。
- 真实消息发送。
- 自动退款、自动赔付、自动售后审核。
- 多页面管理后台。
- 完整运营 BI。

## 7. 核心功能需求

### 7.1 工单队列

客服可以在左侧查看跨境服务工单列表。

每条工单展示：

- 工单标题。
- 消费者名称或脱敏标识。
- 问题类型。
- 工单状态。
- 优先级。
- 风险等级。
- 关联订单号。
- 更新时间。

验收标准：

- 默认选中第一条工单。
- 点击不同工单后，中间详情和右侧 AI 面板同步更新。
- 高风险工单在视觉上清晰可见。

### 7.2 工单详情

客服可以查看当前工单的完整业务上下文。

展示内容：

- 消费者原始问题。
- 历史沟通记录。
- 订单信息。
- 物流轨迹。
- 售后状态。
- 商家信息。
- 相关平台规则摘要。

验收标准：

- 当前工单关联的数据完整展示。
- 不同工单的订单、物流、售后和商家信息不同。
- 页面信息层级清晰，适合快速扫描。

### 7.3 AI 辅助面板

客服可以在右侧查看 AI 辅助结果。

展示内容：

- 服务摘要。
- 问题类型。
- 情绪识别。
- 风险等级。
- 建议处理动作。
- 知识库引用。
- 多语言回复草稿。
- 采纳、编辑、忽略按钮。

验收标准：

- AI 输出与当前工单上下文一致。
- 高风险场景出现人工复核提示。
- 回复草稿不承诺未确认的退款、赔付或具体时效。

### 7.4 顶部指标

工作台顶部展示 Demo 级运营指标。

指标包括：

- 今日服务工单数。
- AI 建议采纳率。
- 高风险工单数。
- 平均处理时长。

验收标准：

- 指标来自 mock 数据或派生统计。
- 指标文案符合平台客服工作台语境。

## 8. 非功能需求

### 可演示性

首页打开后应直接进入核心工作台，不出现默认 Next.js 模板或营销落地页。

### 可维护性

业务类型、mock 数据、AI 辅助逻辑和 UI 组件应分文件组织，避免所有逻辑堆在 `app/page.tsx`。

### 可扩展性

第一阶段虽然使用 mock 数据，但数据结构应贴近真实系统，后续可替换为 API、数据库和真实 AI 调用。

### 安全边界

AI 只能提供辅助建议和回复草稿，不直接执行高风险业务动作。涉及退款、赔付、时效承诺、售后关闭和商家处罚的内容必须提示人工确认。

## 9. 成功指标

### 产品成功指标

- 用户 1 分钟内能理解系统解决的问题。
- 用户能通过切换工单理解不同跨境服务场景。
- AI 面板能清楚体现摘要、风险、建议和回复草稿的价值。

### 工程成功指标

- `npm run lint` 通过。
- `npm run build` 通过。
- 代码结构支持后续替换 mock 数据和 AI 辅助函数。
- TypeScript 类型覆盖核心业务对象。

### 简历成功指标

- 项目能被描述为一个从 0 到 1 的 AI 全栈业务系统。
- 能讲清楚前端工作台、BFF 聚合、数据模型、AI 输出、风险控制和后续架构演进。

## 10. 后续版本规划

### V1：Mock 驱动 Demo

- 完成首页客服工作台。
- 使用 mock 数据和本地 AI 辅助函数。

### V2：服务端聚合

- 增加 API Route 或 BFF 层。
- 将数据聚合逻辑从前端移动到服务端。

### V3：真实 AI 接入

- 接入 AI SDK。
- 使用结构化 JSON 输出。
- 增加失败降级和输出校验。

### V4：数据持久化

- 接入 PostgreSQL 或 MySQL。
- 建立工单、知识库、AI 交互和客服反馈数据表。

### V5：大厂语境工程增强

- 增加 Redis 缓存热点上下文。
- 增加消息队列处理异步 AI 任务。
- 增加审计日志、敏感信息脱敏和运营洞察。
