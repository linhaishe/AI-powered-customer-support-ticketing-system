# 天猫海外 AI 服务中台第一阶段设计文档

## 1. 设计目标

第一阶段目标是把当前 0 功能 Next.js 空壳，开发成一个可演示、可继续迭代的“天猫海外 AI 客服工作台 Demo”。

这个阶段不接入真实天猫海外内部系统，也不直接接数据库和真实 AI。系统先用本地 mock 数据模拟天猫海外现有业务体系，包括消费者、订单、物流、售后、商家、平台规则和客服工单；再用本地 AI 辅助函数模拟摘要、分类、风险识别、处理建议和多语言回复草稿。

最终效果是：用户打开首页后，可以看到一个完整的客服工作台，选择一条跨境服务工单，查看订单、物流、售后和商家上下文，并在右侧看到 AI 辅助处理结果。

第一阶段兼顾两个目标：

- 展示价值：像一个真实可用的天猫海外 AI 客服工作台，而不是静态概念页。
- 保留骨架：代码按真实项目拆分，后续可以替换为 API、数据库和真实 AI 调用。

## 2. 第一阶段范围

### 必做

- 首页改造成客服工作台。
- 提供多条模拟服务工单。
- 支持选择不同工单并刷新详情内容。
- 展示消费者问题、历史沟通、订单、物流、售后和商家上下文。
- 展示 AI 摘要、问题类型、情绪、风险等级、处理建议、知识引用和回复草稿。
- 提供“采纳”“编辑”“忽略”三个 AI 草稿操作入口。
- 把业务类型、mock 数据和 AI 模拟逻辑拆到 `lib/`。
- 把页面 UI 拆到 `components/dashboard/`。

### 暂不做

- 登录和权限系统。
- 真实数据库。
- 真实 AI API 调用。
- 工单创建、保存和服务端持久化。
- 真实天猫海外内部系统接入。
- 自动退款、自动赔付、自动关闭售后等高风险动作。
- 多页面后台、复杂路由和完整管理端。

## 3. 主要演示场景

第一阶段重点打磨一个高质量主场景：**物流清关停滞导致消费者投诉**。

这个场景能集中体现天猫海外业务特点：

- 消费者有明确不满情绪。
- 订单、物流、清关、售后和平台规则都参与判断。
- 客服需要解释跨境物流时效和清关状态。
- AI 可以给出风险提示、处理建议和多语言回复草稿。

同时再准备 2 到 3 条次要工单，用于展示列表切换：

- 消费者咨询退款进度。
- 消费者询问关税或清关费用。
- 消费者反馈商品描述和实物不一致。

## 4. 页面设计

第一阶段只做一个首页工作台，不做营销落地页。

### 整体布局

页面采用运营后台式三栏布局：

- 顶部指标区：展示今日工单、AI 建议采纳率、高风险工单、平均处理时长。
- 左侧工单列表：展示服务工单队列，支持根据状态和风险做视觉区分。
- 中间工单详情：展示当前消费者问题和业务上下文。
- 右侧 AI 辅助面板：展示 AI 理解、建议、引用和回复草稿。

### 顶部指标区

指标用于让 Demo 一进入页面就有“平台工作台”的真实感。

建议指标：

- 今日服务工单
- AI 建议采纳率
- 高风险工单
- 平均处理时长

### 左侧工单列表

每条工单展示：

- 消费者姓名或脱敏标识
- 问题类型
- 简短标题
- 状态
- 风险等级
- 关联订单号
- 更新时间

列表点击后，中间详情和右侧 AI 面板同步刷新。

### 中间工单详情

中间区域是客服判断问题的主区域。

内容包括：

- 消费者原始问题
- 历史沟通摘要
- 订单信息卡
- 物流轨迹卡
- 售后状态卡
- 商家信息卡
- 相关平台规则摘要

### 右侧 AI 辅助面板

右侧是第一阶段的核心价值展示。

内容包括：

- AI 服务摘要
- 问题类型
- 情绪识别
- 风险等级
- 建议处理动作
- 知识库引用
- 多语言回复草稿
- 操作按钮：采纳、编辑、忽略

AI 面板要明确表达：AI 是辅助客服，不是直接替客服做高风险决策。

## 5. 数据设计

第一阶段使用 TypeScript 类型和本地 mock 数据。数据结构需要像真实业务对象，方便后续替换成数据库表或接口返回。

建议核心类型：

```ts
type ServiceCase = {
  id: string;
  title: string;
  status: "new" | "in_progress" | "waiting_consumer" | "escalated" | "resolved";
  issueType: string;
  priority: "low" | "medium" | "high" | "urgent";
  riskLevel: "low" | "medium" | "high";
  consumerId: string;
  orderId: string;
  afterSalesId?: string;
  merchantId: string;
  messages: CaseMessage[];
  updatedAt: string;
};
```

```ts
type AiAssistantResult = {
  summary: string;
  issueType: string;
  sentiment: "calm" | "confused" | "dissatisfied" | "angry";
  riskLevel: "low" | "medium" | "high";
  suggestedActions: string[];
  replyDraft: string;
  citations: KnowledgeArticle[];
};
```

建议 mock 数据覆盖：

- `serviceCases`
- `consumers`
- `orders`
- `logisticsTraces`
- `afterSalesCases`
- `merchants`
- `knowledgeArticles`

## 6. 模块设计

建议代码结构：

```txt
app/
  page.tsx
  globals.css

components/
  dashboard/
    AiAssistantPanel.tsx
    CaseDetail.tsx
    CaseList.tsx
    ContextCard.tsx
    MetricsBar.tsx
    StatusBadge.tsx

lib/
  ai-assistant.ts
  mock-data.ts
  types.ts
```

### `lib/types.ts`

集中定义业务对象和 AI 输出类型。

作用：

- 让 mock 数据、组件和 AI 模拟函数共享同一套结构。
- 后续接数据库或 API 时减少重写。

### `lib/mock-data.ts`

提供第一阶段所有模拟业务数据。

作用：

- 模拟天猫海外现有系统。
- 让前端页面不依赖真实后端。
- 支持多个工单切换演示。

### `lib/ai-assistant.ts`

提供本地 AI 辅助函数。

作用：

- 根据当前工单和上下文返回结构化 AI 辅助结果。
- 第一阶段可以使用规则或静态映射。
- 后续可以替换成真实 AI SDK 调用。

### `components/dashboard/*`

负责工作台 UI 展示。

组件边界：

- `MetricsBar`：顶部指标。
- `CaseList`：工单列表和选中态。
- `CaseDetail`：当前工单详情。
- `ContextCard`：订单、物流、售后、商家等上下文卡片。
- `AiAssistantPanel`：AI 辅助结果和操作按钮。
- `StatusBadge`：状态、风险、优先级标签。

## 7. 数据流

第一阶段数据流保持简单：

1. `app/page.tsx` 从 `mock-data.ts` 读取工单和业务上下文。
2. 页面维护当前选中的 `caseId`。
3. 根据 `caseId` 找到当前工单、消费者、订单、物流、售后、商家和知识库。
4. 调用 `buildAiAssistantResult()` 获取 AI 辅助结果。
5. 把数据传给三栏组件展示。
6. 点击“采纳”“编辑”“忽略”时，只在前端展示轻量反馈状态，不持久化。

这种设计能让第一版快速成型，同时保持未来演进路径：

- `mock-data.ts` 可以替换成 API 请求。
- `buildAiAssistantResult()` 可以替换成 AI SDK 调用。
- 前端组件基本不用大改。

## 8. 交互设计

### 工单切换

用户点击左侧工单后：

- 左侧选中态变化。
- 中间详情切换。
- 右侧 AI 辅助结果切换。

### AI 草稿操作

用户可以对回复草稿执行：

- 采纳：显示已采纳状态，表达 AI 建议被使用。
- 编辑：让草稿进入可编辑状态或展示编辑入口。
- 忽略：显示已忽略状态，表达客服保留人工判断。

第一阶段不需要真正发送消息，也不需要保存反馈到后端。

### 风险提示

高风险工单需要明显提示，但不应阻断操作。

示例：

- “涉及赔付风险，建议主管复核。”
- “物流轨迹长时间未更新，建议升级物流专员。”
- “消费者多次表达不满，回复需谨慎承诺时效。”

## 9. 错误和空状态

第一阶段虽然是 mock 数据，也需要保留基础异常处理思路：

- 没有选中工单时，默认选中第一条。
- 找不到订单或物流上下文时，显示“暂无关联数据”，而不是页面报错。
- AI 结果不可用时，展示“AI 辅助暂不可用，请按 SOP 人工处理”。

这些状态可以先在代码里准备分支，不一定都要做复杂 UI。

## 10. 测试与验证

第一阶段验证重点：

- 页面能正常启动。
- 工单列表可以切换。
- 不同工单展示不同详情。
- AI 面板内容和当前工单一致。
- 响应式布局在桌面宽度下可读、不卡顿、不重叠。
- `npm run lint` 通过。
- `npm run build` 通过。

如果后续加测试，可以优先补：

- `buildAiAssistantResult()` 的输入输出测试。
- 业务类型和 mock 数据完整性测试。

## 11. 后续演进

第一阶段完成后，建议按以下顺序迭代：

1. 增加 API Route，把 AI 辅助函数从页面中解耦。
2. 接入真实 AI SDK，输出结构化 JSON。
3. 加入 Prisma 和 PostgreSQL，持久化工单、知识库和 AI 交互记录。
4. 增加 AI 反馈记录：采纳、编辑、忽略、错误原因。
5. 增加知识库管理页。
6. 增加运营洞察页。
7. 增加权限、审计和敏感字段脱敏。

## 12. 第一阶段成功标准

第一阶段完成后，应达到以下标准：

- 打开首页即可看到一个完整的天猫海外 AI 客服工作台。
- 用户能通过切换工单理解不同跨境服务问题。
- 每条工单都有订单、物流、售后、商家和规则上下文。
- AI 面板能清楚展示摘要、风险、建议和回复草稿。
- 代码结构支持后续替换 mock 数据、接真实 AI 和接数据库。

如果一个观看 Demo 的人能在 1 分钟内理解“AI 如何帮助天猫海外客服处理复杂跨境问题”，第一阶段就是成功的。
